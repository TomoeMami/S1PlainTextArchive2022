前端仔，过去大半年写了不少nodejs转发层和访问量不高的CRUD接口。

作为开发，我觉得靠某种技术吃饭却对原理所知甚少，这种体验非常差。所以晚上有空就看nodejs源码，写写注释，参考了很多资料。希望能巩固cs基础，也更了解nodejs。

js层自然没啥问题，虽然很绕。libuv写demo+调试，问题也不大。中间的c++比较头疼。v8可参考的文章不多，c++ binding就更少了。

之前自作聪明学rust，写了几个cli。实在缺乏应用场景，就没再写下去。至于deno，在可见的未来，感觉公司项目不会大规模应用。

c++ binding部分边看边查语法。有些如智能指针还好，有些试了很多关键字组合才能搜到。这种情况很少，但感觉很糟糕。

转码是从c++开始的。当时毫无cs专业背景 + 国内教材，简直看是天书。现在补了不少cs基础，加上源码里函数、变量和类见名知意，虽然勉强能看懂主线流程，但提到c++还是莫名的恐惧😫。

C++加的特性，可以不用，就像C++里面包含了大部分C的特性，很多也可以不用

别人写的代码，大部分可以不看，就像你写C的程序不看Java的代码一样，没有用了C++就得看所有C++代码的道理

C语言特性并不少，特别是undefined behavior满天飞，要不要和Basic, Pascal, Fortran什么的比较一下？

像下面这种代码，其他语言怕是没几个能写的

static __inline long __syscall4(long n, long a1, long a2, long a3, long a4)

{

        unsigned long ret;

        register long r10 __asm__("r10") = a4;

        __asm__ __volatile__ ("syscall" : "=a"(ret) : "a"(n), "D"(a1), "S"(a2),

                                                  "d"(a3), "r"(r10): "rcx", "r11", "memory");

        return ret;

}

要说特性少，那是因为C设计的时候受到编译器实现的影响太大了

1)缺少内存安全、异常处理

内存安全和异常处理这两个结合起来，造成很多时候一个函数里面有一半的代码在判断出错的时候怎么释放内存，很容易写错，当然有些人说可以用goto跑到函数最后去统一处理，但是这导致各种重复判断，还是很容易出错

2)缺少泛型

泛型也是一个无法回避的问题，C里面就没有比较正常的数据结构封装(列表、双链表、散列表、红黑树)，实现上有两个都不能让人满意的做法：1)用宏去黏合代码 2)用代码生成器。总之就是很难用。这个问题在Google的Go里面也是非常严重的，于是他们做了Go 2

3)缺少多线程标准

再说一下C的多线程的问题：C一直缺少标准线程库，atomic相关的函数也是C11才加入(不过似乎使用还不广泛)，导致要用C写多线程程序，不仅需要了解语言特性，还需要了解操作系统和编译器

4)缺少异步处理构造

说了多线程就不得不说回调管理了，C语言甚至不支持lambda函数，更别说coroutine了，建议需要写异步程序的远离C(和C++)。不要说lambda是什么新特性，Lisp可比C老得多

5)#include

这个就不多说了，没见过其他主流语言有这个功能的。有点常识的人写的编译器都不会这么做

6)缺少字符串处理

字符串用一个bug一样的没有长度的\0结尾的编码不明的指针，不知道造成了多少内存越界攻击漏洞。编码不明，造成了多少程序乱码。C++虽然也不行，但是至少在尝试，std::u8string这种比没有好

那么，缺少了这么多特性，要怎么"容易地"写正确的通用跨平台多线程异步程序呢？还是说就只是"很困难地"写写操作系统内核、驱动程序？选择困难比没得选择更容易客服，选择困难只需要学习什么是正确的选择，而没得选择就是没得选择。当然你要fork一个C编译器是没问题的，但是也要有人用

看看现在llvm的新C标准库是用C++写的，Linux也开始引入Rust了哦

现在C语言的地位，完全是基于历史原因。主流操作系统的接口基于C(Win32和POSIX)，而除了C，一般只有汇编可以用于调用这些接口。在Basic, Pascal, Fortran还流行的时候，写编译器的时候写写汇编调用操作系统接口那是轻松加愉快的。但随着时间的推移，后来的语言越来越依赖于这些C接口。于是大家投入了很多精力去维护一个陈旧的语言

C++有很多缺点，但是和C的缺点是有很大差异的，但是我也可以列一下

1)模板和#include导致编译速度慢

现在的模板是个大毒瘤。模板里面的代码在每个展开的地方都需要编译，导致C++的项目只要上一点规模就编译速度特别慢，特别是在#include加持下

模板参数还能进行各种加减乘除什么的，导致编译器写起来特别复杂，经常出现编译器内部错误

但是模板还是不得不用，因为这是当前实现泛型的唯一路径。(代码生成器的做法除外)

模板还有一个问题是经常报莫名其妙的错误，这个引入了concept

解决#include的module，现在实现还不完整，且有很多问题

2)面向对象

C++的继承体系是常见语言里面最复杂的，有多类继承、虚继承、私有继承等各种独有机制。实际上继承并非写程序的必要机制。最近10年C++的标准应该基本上没有碰这块。不用问题不大

3)标准库匮乏

网络库(socket、http)还没有标准化

字符串编码转换没有标准化

4)包管理？
